Checking the web server :

WEB ENUMERATION:

Using Gobuster we can brute-force URIs including directories and files :

1

gobuster **dir** -w /home/kali/Downloads/SecLists-master/Discovery/Web-Content/raft-medium-directories.txt -u [http://10.10.77.2/](http://10.10.77.2/) **\-s** "204,301,302,307,401,403" **\-x** php,py,sh,js,txt

-dir to run gobuster as directory/file enumeration mode as there is also vhost for VHOST enumeration mode.

-w is for the wordlist , I used the SecLists one that you can get from here : https://github.com/danielmiessler/SecLists

-s for status-codes that we want to see back

-x for extentions we are looking for ( at this point we are looking for anything)

We got back results :

/blog

/javascript

/phpmyadmin

/wordpress

We now have to investigate eash directory manually and see if we can find any outdated or missplaced systems.

Before we can reach any directory we need to add the domaine name to our hosts file .

1

sudo nano /etc/hosts

And add :

“10.10.77.2” internal.thm «Don’t forget to use your given address»

/blog :

Immediatly using the Wappalyzer or with the recent finding on /wordpress we can see that it’s using WordPress.

And a an outdated version of it.

We can use wpscam (a tool to scan wordpress websites for exploits)

Let’s no rush into rabbit holes and keep digging.

II-FINDING THE VULNERABLE PIECE.

/blog is a blog website with a “Hello World!” first blog and a search query.

The article contains nothing of interest but the search function can be exploited :

Maybe we can fuzz the s paramater and get an /etc/passwd through an LFI.

Before we dive into another rabbit hole let’s keep investigating until we run out of ressoruces.

At the bottom of the page we have a login button :

That takes us to /wp-login.php

But we need credentials for this admin portal.

Using random creds for this results in an error :

But a very weird error : Unknown username.

Let’s try admin , wpadmin , root …

The username admin got us a new error that confirms there is an admin username , all we have to do now is find the password.

We can use the same wpscan tool to bruteforce a password since we have a valid username.

1

wpscan --url [http://internal.thm/blog/wp-login.php](http://internal.thm/blog/wp-login.php) --usernames admin --passwords /usr/share/wordlists/rockyou.txt

The password is not sexyme but we are trying at least.

And we got a match after 6 long minutes.

Now that we have the password and username we can access the WordPress admin portal.

III-PRIVESC.

aaaand the jenkins.txt as our endless persuit of power never ends.

We have a jenkins service running on the machine so we have to tunnel it in a way to our machine so we can investigate it.

“Jenkins is an open source continuous integration/continuous delivery and deployment (CI/CD) automation software DevOps tool written in the Java programming language”

[

How to Use SSH Tunneling to Access Restricted Servers and Browse Securely

An SSH client connects to a Secure Shell server, which allows you to run terminal commands as if you were sitting in…

www.howtogeek.com

](https://www.howtogeek.com/168145/how-to-use-ssh-tunneling/)

Using SSH tunneling I can simply establish an SSH connection with the SSH server and tell the client to forward traffic from a specific port from my local host (localhost:5555) to the address of the service and its port.

1

ssh -L 5555:172.17.0.2:8080 aubreanna@10.10.77.2

Now we can access the jenkins servcie on our localhost:5555

This late at night I will just brute force it using hydra and hope for the best.

This is a POST request to /j_acegi_security_check with the parameters :

j_username=azer&j_password=azer

This is the full request as cURL.

Using Hydra :

1

hydra 127.0.0.1 -s 5555 -V -f http-form-post “/j\_acegi\_security\_check:j\_username=^USER^&j\_password=^PASS^&from=%2F&Submit=Sign+in&Login=Login:Invalid username or password” -l admin -P /usr/share/wordlists/rockyou.txt
  • ^USER^: tells Hydra to use the username or list in the field.
  • ^PASS^: tells Hydra to use the password list supplied.
  • -l: indicates a single username
  • -P: indicates use the following password list e.g. rockyou.txt

Using the fresh creds we found we can get to the jenkins service.

Googling ways to exploit Jenkings to get a reverse shell I have found this Script Console exploit under Manage Jenkings Tab.

https://www.n00py.io/2017/01/compromising-jenkins-and-extracting-credentials/

Here we can execute a groovy script.

Apache Groovy is a Java-syntax-compatible object-oriented programming language for the Java platform. It is both a static and dynamic language with features similar to those of Python, Ruby, and Smalltalk.

Using the dear pentestmonkey we can get a simple java payload for reverse shell :

[

Reverse Shell Cheat Sheet

If you’re lucky enough to find a command execution vulnerability during a penetration test, pretty soon afterwards…

pentestmonkey.net

](https://pentestmonkey.net/cheat-sheet/shells/reverse-shell-cheat-sheet)

And we managed to get a reverse shell from the jenkins service.

Now we look inside the container and we can find in the /opt file a note containing the root ssh creds.

Now we just SSH to the machine with the freshly obtained creds and we are ROOT.
